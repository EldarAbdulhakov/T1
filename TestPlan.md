# План тестирования

## 1. Введение
### 1.1 Цель тестирования
Проверить корректность работы конечной точки API для получения статуса продукта.

Проверить корректную обработку:
* параметров запроса (обязательных и необязательных)
* авторизации (authToken)
* обработки несуществующих и некорректных данных
* формата и содержимого ответа (JSON)
* кодов ответов HTTP

### 1.2 Персонал
Один тестировщик (100% загруженность за все время проекта). Время: 14 рабочих дней (112 рабочих часов)

## 2. Объект тестирования
Метод GET /products/{productid}/status?authToken=&recalculate=&owner=&region=

## 3. Аппаратные и программные требования
* Программное обеспечение: Windows 11
* Аппаратное обеспечение: Процессор Intel(R) Core(TM) i3-2310M CPU @ 2.10GHz, 16 Gb RAM

## 4. Ресурсы тестирования
* Коммуникация (Telegramm)
* Документация (GitHub)
* Тестирование (Postman)
* Скриншоты (Icecream Screen Recorder 7)

## 5. Типы проверок
* Позитивные тесты
* Негативные тесты
* Тестирование границ
* Тестирование безопасности
* Тестирование производительности

## 6. Критерии начала тестирования
* Наличие требований для объекта тестирования
* Зафиксирована сборка API для проведения тестирования
* Готовность тестовой среды
* Доступность инструментов тестирования
* Утвержден тест-план и тест-кейсы

## 7. Критерии приостановки тестирования
* Обнаружен дефект с Severity Critical или Blocker
* Непредвиденные проблемы с доступностью API или тестовой среды
* Изменение требований или функциональности

## 8. Критерии возобновления тестирования 
* Дефект Critical или Blocker исправлен
* Исправлен доступ к API или тестовому стенду
* Тест-кейсы обновлены для изменненных требований

## 9. Критерии завершения тестирования
* Функционал протестирован: пройдены все тестовые случаи
* Дефекты выявлены и задокументированы
* Проведено ревью и анализ результатов тестирования

## 10. Риски
* Недоступность тестового стенда - необходимо немедленно сообщить ответственным за состояние тестового стенда сотрудникам для оперативного решения вопроса. Использовать резервный тестовый стенд.
* Не готова тестовая документация - начать тестирование с приоритетных и критичных случаев в формате исследовательского тестировани. Параллельно оформлять и дополнять тест-кейсы.
* Время: Заказчик указал крайний срок 17.10. Рекомендуется приложить все усилия, чтобы завершить проект к 15.10, чтобы один день оставался доступным для любых неожиданных проблем
* Отключение электричества
* Заболевание тестировщика
